# Phase 07 - UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase UI Scope

Phase 7 must enrich the existing harness-neutral triage workspace with shared read-only repository truth, optional shared read-only GitHub context, and portable archive workflows without turning Agent Workbench into a git client, GitHub client, or archive-management app.

Primary visual anchors by surface:

- `Projects`: the selected project row and its repo-truth panel become the main place to inspect branch, HEAD, dirty state, change counts, remote, PR summary, and export availability.
- `Run Audit`: the `Git / GitHub` evidence group becomes a real audit surface with shared snapshot truth and explicit reasons when repo context could not be collected.
- `Data Sources`: imported archives appear as first-class read-only sources with explicit source-kind metadata, archive-specific states, and a focused `Import Archive` action.

Keep V1 and the Phase 7 boundary intact. Do not add clone, checkout, commit, fetch, merge, rebase, branch cleanup, PR creation, PR comment actions, review submission, terminal control, arbitrary shell execution, diff-body viewers, full GitHub conversation timelines, archive editing, archive rebinding to live roots, or raw-artifact export enabled by default.

Phase 7 must preserve the Phase 6 truth rule: unsupported or missing repo evidence stays explicit. Never flatten missing git or GitHub data into `Clean`, `Passed`, `0`, or an empty section.

Imported archives remain read-only sources. They may display archived snapshot data that was already captured in the archive, but they must never trigger host-side validate, scan, watch, git, or GitHub operations after import.

## Layout Contract

| Area | Contract |
|------|----------|
| Window | Preserve the existing desktop shell baseline. Full repo-context density is optimized for 1100px by 720px and above, and remains usable down to the current 800px by 680px minimum by stacking detail panels beneath lists. |
| Shell chrome | Preserve the current shell: 224px left sidebar, 56px topbar, one border divider, and route-first navigation. No new global tabs, drawers, or utility rails. |
| Navigation | Keep the Phase 6 route map: `/overview`, `/projects`, `/data-sources`, `/sessions`, `/sessions/:sessionId`, `/sessions/:sessionId/run-audit`, and `/diagnostics`. Phase 7 adds depth inside `Projects`, `Run Audit`, and `Data Sources`; it does not create a dedicated GitHub page or archive-only route. |
| Projects layout | Use the Phase 6 split list/detail pattern as the primary Phase 7 visual language. Left list column remains `minmax(360px, 42%)`; right detail panel is `minmax(0, 1fr)`. The detail panel is sectioned into `Repository`, `Changes`, `GitHub`, and `Archive Export`. |
| Project rows | Project rows remain dense bordered rows, not decorative cards. Each row should surface project name, repo path, observed harnesses, latest activity, latest audit/verification, branch summary, dirty-state summary, and PR availability in one scan-first band. Row height is at least 96px. |
| Project detail panel | Lead with project name, repo path, and observed harnesses. Below that, show validated git fields, then GitHub snapshot summary, then export controls. Keep warnings inline near the affected field group instead of collecting everything into one bottom-only alert. |
| Run Audit layout | Preserve the current sticky verdict rail plus grouped evidence sections. The `Git / GitHub` group becomes a real bordered section that can show validated repo root, branch, HEAD, dirty state, counts, remote URL, PR summary, check summary, review or merge status, and explicit reasons when data is unavailable. |
| Data Sources layout | Preserve the existing list/detail management surface. Imported archives appear in the same source list, not in a separate archive browser. The detail panel adds `Source Kind`, `Added By`, archive metadata, read-only rules, and `Import Archive` actions. |
| Archive source rows | Imported archive rows use the same bordered-row treatment as local sources, but include a quiet `Imported Archive` or `Read Only` chip. They must not visually resemble an error state by default. |
| Export flow | Export starts from an explicit user-triggered action on the selected project or current run audit view. Use a compact confirmation surface with archive scope summary, an opt-in raw-artifact checkbox, privacy warning copy, and a single export action. Avoid wizard-style multi-step navigation. |
| Import flow | `Import Archive` opens a main-owned archive picker or equivalent main-owned file selection flow. After import, the new source appears in the existing list and becomes the selected detail panel. Do not add drag-and-drop-only import or a renderer-local file parser. |
| Dialogs and sheets | Use compact, single-purpose confirmation surfaces only for export privacy confirmation and any import failure explanation that needs more than one line. Do not move project or archive inspection into modal-first flows. |
| Empty/error areas | Center within the route content area using `2xl` top offset. Use restrained operator-style copy with one next action. No marketing tone, no hero illustrations, no celebratory success screens. |

## State Vocabulary

Use these exact user-facing labels unless an upstream DTO supplies a more specific sanitized reason:

| Category | Labels |
|----------|--------|
| Git snapshot | `Available`, `Unknown`, `Unsupported` |
| GitHub snapshot | `Available`, `Unknown`, `Unsupported`, `No Matching PR` |
| Repo cleanliness | `Clean`, `Dirty`, `Unknown`, `Unsupported` |
| Archive source kind | `Imported Archive`, `Local Source` |
| Archive mutability | `Read Only` |
| Export raw-data choice | `Include Raw Artifacts`, `Normalized Only` |
| Existing audit truth | `Active`, `Cancelled`, `Verification Failed`, `Incomplete`, `Needs Review`, `Clean`, `Unknown` |
| Existing verification truth | `Passed`, `Failed`, `Not Run`, `Unsupported`, `Unknown` |

Truth rules:

- Git or GitHub sections appear whenever the concept applies to the current surface, even when data is unavailable.
- `Unknown` means the provider could not produce trustworthy data from validated inputs, timed out, or encountered an indeterminate condition.
- `Unsupported` means the current source or adapter cannot represent that capability at all.
- `No Matching PR` is a neutral GitHub outcome, not an error and not `Unknown`.
- Imported archives may display archived git or GitHub snapshot values if they are already part of the archive. If the archive lacks that data, show `Unknown`; do not re-query the host repo or network.
- Change counts are shown only when the underlying git snapshot is present. Never show `0` as a substitute for unavailable repo data.

## Interaction Contract

| Interaction | Contract |
|-------------|----------|
| Projects initial load | Keep the existing loading and error model, but when data loads successfully the route must render real repo-context columns and a selected detail panel rather than Phase 6 placeholders alone. |
| Project selection | Clicking a project row updates the right-hand repo-truth panel in place. Keyboard navigation remains list-first and must preserve the selected project after reload when it still exists. |
| Repo summary | The selected project panel shows repo path, branch, HEAD, dirty state, changed files, untracked files, additions/deletions, and remote URL when available. If the project root could not be validated, keep the section visible and explain why git data is unavailable. |
| GitHub summary | When a validated git snapshot and remote URL exist, show PR summary, checks summary, and review or merge status if available. If `gh` is unavailable, auth is missing, the request times out, or no PR matches, show a truthful field-level state and concise helper text. |
| Run Audit git evidence | The `Git / GitHub` evidence group uses the same shared snapshot vocabulary as Projects. It must not invent session-local git truth from shell text when the shared provider did not attach a snapshot. |
| Data source import | `Import Archive` is available from `Data Sources` and registers the selected archive as a persistent read-only source. The renderer never reads the archive file directly. |
| Imported source detail | Imported archive sources show `Source Kind`, `Added By`, archive path or label, session/artifact counts when present, and a clear `Read Only` rule. Validate, scan, rescan, watch, git, and GitHub actions stay unavailable for imported archives. |
| Export entrypoints | Provide `Export Project Archive` on the selected project panel and `Export Session Archive` on Run Audit. These are explicit, user-triggered, and never run automatically during reload or scan flows. |
| Export confirmation | Before writing an archive, show scope summary, default `Normalized Only`, optional `Include Raw Artifacts` checkbox, and a privacy warning explaining that transcripts, sidecars, repo paths, and command output may contain sensitive data. |
| Raw opt-in | Raw-artifact export is opt-in only. The checkbox is unchecked by default. If raw artifacts are unavailable for the current archive scope, render the option as disabled with a reason instead of hiding it. |
| Import completion | After a successful import, focus returns to `Data Sources`, the new archive source is selected, and the detail panel explains that imported sessions render from archived data only. Do not auto-navigate to Projects or Sessions. |
| Import failure | Import failures stay scoped to the import flow and use sanitized copy. Do not create partial writable sources or silently ignore failed imports. |
| Read-only rule | No UI control in Phase 7 may mutate repositories, remote pull requests, live source roots, or imported archive contents. Export writes a new archive file only; import registers a read-only source only. |
