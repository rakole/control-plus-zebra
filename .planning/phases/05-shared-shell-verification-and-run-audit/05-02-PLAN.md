| Boundary | Description |
|----------|-------------|
| parsed shell summaries -> verification result | Verification must rely on shared shell truth instead of tool success, UI heuristics, or adapter-specific claims. |
| capability snapshots -> verification headline | Missing or unsupported shell capability must remain explicit and must never flatten into passed or clean states. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02-01 | Repudiation | verification classification | mitigate | Derive verification only from persisted shared shell summaries and keep supporting command IDs/reason codes for later audit traceability. |
| T-05-02-02 | Tampering | capability-gap handling | mitigate | Resolve capability truth in session -> source -> adapter order and map blocked conclusions to `unsupported` or `unknown`, never `passed`. |
| T-05-02-03 | Elevation of Privilege | verification headline | mitigate | Limit qualifying verification intents to `test`, `build`, `typecheck`, and `lint` so install/git/other commands cannot silently act as verification proof. |
| T-05-02-SC | Tampering | npm/pip/cargo installs | mitigate | No package install is planned. If implementation attempts any install, stop and add a blocking human verification checkpoint because slopcheck was unavailable. |
</threat_model>
