<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| verification result + lifecycle evidence -> run audit | The audit headline must remain conservative when signals conflict and must never silently flatten cancellation, parser warnings, or missing evidence into clean success. |
| internal derived cache payload -> IPC session summaries | Richer audit internals must not leak into public view models before Phase 6 owns presentation and capability-gated UI behavior. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03-01 | Repudiation | run audit engine | mitigate | Persist both the primary status and attention reasons with supporting IDs so claim-vs-evidence disputes remain traceable. |
| T-05-03-02 | Tampering | clean classification | mitigate | Gate `clean` behind explicit supported git context and use `needs-review` plus `capability-missing` when git evidence is unavailable. |
| T-05-03-03 | Information Disclosure | IPC session previews | mitigate | Preserve the existing sanitized boundary and assert in tests that run-audit conclusion fields do not appear in current session summaries/previews. |
| T-05-03-SC | Tampering | npm/pip/cargo installs | mitigate | No package install is planned. If implementation attempts any install, stop and add a blocking human verification checkpoint because slopcheck was unavailable. |
