# Phase 03 - UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase UI Scope

Phase 3 must add a quiet, work-focused Harnesses and Data Sources settings surface for source-root configuration, validation, explicit scan/rescan, cache state, compact diagnostics, enabled/disabled source state, and watcher support truth.

The route must use a split list/detail management surface. The list makes source health scannable at a glance. The detail panel owns add/edit/validate/rescan interactions and concise operational diagnostics.

Keep Phase 6 out of scope. Do not build Overview dashboards, full Diagnostics pages, Session Detail timelines, Run Audit panels, deep cache inspectors, native directory pickers, full live watcher controls, or provider-specific UI branches. V1 remains read-only with respect to harnesses and repositories: no session launching, terminal control, approve/reject, PR creation, cleanup, arbitrary shell execution, or source-root filesystem mutation.

Renderer behavior must stay harness-neutral and consume typed IPC view models only. Unsupported or unknown capability, watcher, cache, validation, or scan states must render as explicit `Unsupported` or `Unknown` labels, never as `0`, `Passed`, `Clean`, or hidden values.

Primary visual anchor: the selected data source row and its detail panel. Route title, source list summary, and source actions are secondary.

| Token | Value | Usage |
|-------|-------|-------|
| Text strong | `#111827` | Primary readable text |

State color rule: successful and cached states use neutral outlined badges with explicit text, not green success styling. Unsupported, unknown, stale, disabled, and never-scanned states use neutral badges. Failed validation, failed scan, and route load errors may use destructive red.

## Layout Contract

| Area | Contract |
|------|----------|
| Window | Minimum useful viewport is 1024px by 680px. Content must remain usable down to 800px width by stacking the detail panel below the list. |
| Shell chrome | Preserve the Phase 2 shell: persistent left sidebar at 224px on desktop, 56px topbar, secondary surface, and single border divider. |
| Navigation | Show Overview, Projects, Data Sources, Sessions, and Diagnostics. Data Sources and Sessions are navigable. Data Sources is active on `/data-sources`. Overview, Projects, and Diagnostics remain disabled placeholders with muted styling and `aria-disabled="true"`. |
| Route | Add `/data-sources` with route title `Data Sources`. Phase 3 may make `/data-sources` the default route while preserving `/sessions`. |
| Data Sources layout | Desktop uses two columns: source list `minmax(360px, 42%)` and detail panel `minmax(0, 1fr)` with a 32px gap. At widths below 960px, stack list above detail. |
| Source list | Render sources as dense rows separated by borders, not decorative cards. Row height is at least 88px. Selected row has a 4px accent left border and `#EFF6FF` fill. |
| Source row content | Each row shows adapter display name, source display name or root path, enabled/disabled state, validation status, latest scan/cache status, diagnostic count, and watch support state. Long paths truncate with ellipsis in the row but show the full value in the detail panel and title tooltip. |
| Detail panel | Render selected source title, adapter metadata, enabled switch, display name input, typed root path input, validation summary, scan/cache summary, watch support truth, compact diagnostics list, and source actions. Use section dividers, not nested cards. |
| Add/edit form | Use a single detail-panel form. Adapter select appears first, then optional display name, then root path. Do not open a wizard or modal for the base flow. |
| Diagnostics list | Show compact diagnostic rows with severity, code, short message, and source area (`source`, `adapter`, `cache`, or `normalization`). Limit visual depth to the selected source; do not build the full Diagnostics route. |
| Empty/error areas | Center within the route content area using `2xl` vertical offset. Do not use marketing hero layout, oversized illustration, or explanatory feature prose. |

## State Vocabulary

Use these exact user-facing labels unless an upstream view model supplies a more specific diagnostic message:

| Category | Labels |
|----------|--------|
| Source enabled state | `Enabled`, `Disabled` |
| Validation | `Not Validated`, `Validating`, `Valid`, `Validation Failed`, `Unsupported`, `Unknown` |
| Scan/cache | `Never Scanned`, `Scanning`, `Scan Failed`, `Scanned with Diagnostics`, `Cached`, `Stale`, `Unsupported`, `Unknown` |
| Watch support | `Watch Supported`, `Watch Unsupported`, `Watch Unknown` |
| Capability truth | `Supported`, `Unsupported`, `Unknown` |

Never show missing diagnostics, unsupported scanning, unsupported watching, or unknown cache state as zero activity or a clean source.

## Interaction Contract

| Interaction | Contract |
|-------------|----------|
| Initial load | Show a compact skeleton with three source rows and one detail block while data sources are loading. |
| Source selection | Clicking a row selects it and updates the detail panel in place. Arrow keys move through rows; Enter selects the focused row. Preserve the selected source after reload when it still exists. |
| Add Source | `Add Source` creates a draft detail form, focuses the typed path input after adapter selection is available, and does not scan automatically. |
| Typed path entry | Use a text input only in Phase 3. Do not add a native directory picker. Empty path disables `Validate Source`. The renderer must not expand paths, read files, or check existence locally. |
| Adapter selection | Populate from adapter metadata returned by IPC. Renderer code may display adapter names and capability metadata, but must not branch on provider IDs except labels and capability gates. |
| Validate Source | `Validate Source` validates the selected adapter/path through the preload bridge. It does not scan. Validation failure preserves the attempted source entry and shows diagnostics. Successful validation enables the scan action. |
| Edit source | Editing adapter or path marks validation as `Not Validated` or `Stale` per view model and disables scan until validation succeeds again. Do not silently drop prior diagnostics; move them under the latest validation/scan summary. |
| Scan Source | `Scan Source` is available only when the source is enabled and validation is `Valid`. It explicitly runs shared scanner orchestration through IPC and updates scan/cache state. |
| Rescan Source | After a source has scanned at least once, label the scan action `Rescan Source`. It must remain explicit and user-triggered; no auto-rescan on edit. |
| Enable/Disable | The enabled switch persists local source registry state only. Disabling a source does not delete registry data, clear cache, mutate the source root, or imply clean status. Disabled sources cannot scan but can still be selected and inspected. |
| Cache stale state | Show `Stale` when artifact/index/cache inputs changed or are indeterminate. Provide a concise reason from diagnostics when available and offer `Rescan Source` only if validation remains `Valid` and the source is enabled. |
| Watch support | Show watch-plan support truth as read-only metadata. Do not expose full live watcher controls in Phase 3. |
| Diagnostics | Show selected-source diagnostics inline. Use sanitized messages. Do not expose raw stack traces, raw filesystem dumps, adapter-private object names, or shell command details. |
| Disabled nav | Placeholder nav entries do not navigate. Use tooltip copy `Available in a later phase`. |
| Error handling | Route load errors show sanitized user-facing copy and a `Reload Data Sources` action. Per-source errors remain scoped to the selected source detail panel. |
| Read-only scope | No UI controls for launch, approve, reject, terminal input, PR creation, cleanup, delete source, reset source, clear cache, or arbitrary shell execution are allowed in Phase 3. |
