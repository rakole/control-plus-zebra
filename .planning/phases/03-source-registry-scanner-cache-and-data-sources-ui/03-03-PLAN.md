<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| normalized scanner output -> file-backed cache | Only validated normalized data should be persisted. |
| cache files -> main-process view models | Cache files may be missing, corrupt, stale, or tampered with. |
| source summaries -> renderer DTOs | Cache/source truth must not leak raw filesystem data or imply success from missing evidence. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Tampering | `file-backed-cache-store.ts` | mitigate | Validate cache files with strict schemas and diagnose corrupt or schema-mismatched payloads as stale/unknown. |
| T-03-03-02 | Spoofing | `cache-keys.ts` | mitigate | Include adapter ID, source ID, artifact/native identity, versions, schema, metadata, and diagnostics hash in keys. |
| T-03-03-03 | Information Disclosure | `session-view-model-service.ts` | mitigate | Continue returning sanitized DTOs only; never return raw artifact paths, raw events, stack traces, or adapter-private object names. |
| T-03-03-04 | Denial of Service | cache reload path | accept | The file-backed cache is scoped to local fixtures/source roots; large-volume indexing remains outside this phase until realistic volume proves SQLite need. Keep operations deterministic and test-temp-dir injectable. |
| T-03-03-SC | Tampering | npm/pip/cargo installs | mitigate | No package install is planned. If implementation attempts any install, stop and add a blocking human verification checkpoint because slopcheck was unavailable. |
</threat_model>
