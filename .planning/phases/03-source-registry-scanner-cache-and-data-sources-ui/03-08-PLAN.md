<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| source/scanner/cache services -> Data Sources DTOs | Internal source records and diagnostics become renderer-facing view models. |
| user-entered source requests -> service methods | User-entered adapter/path/source actions must be validated before they mutate local app registry/cache state. |
| main service -> renderer DTOs | Diagnostics and paths must be sanitized and capability truth must not be flattened. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-08-01 | Tampering | Data Sources request DTOs | mitigate | Validate all requests with strict Zod schemas before service operations update local registry/cache state. |
| T-03-08-02 | Information Disclosure | `data-sources-view-model-service.ts` | mitigate | Return sanitized diagnostics and DTOs only; no raw stack traces, raw filesystem dumps, raw events, or adapter-private object names. |
| T-03-08-03 | Repudiation | scan/validation state | mitigate | Persist validation, scan, cache, diagnostics, and timestamps so failures and stale state remain visible. |
| T-03-08-04 | Spoofing | capability/watch/cache labels | mitigate | Carry explicit unsupported/unknown/stale/failed states through DTO schemas rather than deriving clean labels from counts. |
| T-03-08-SC | Tampering | npm/pip/cargo installs | mitigate | No package install is planned in this plan. If implementation attempts any install, add a blocking human verification checkpoint because slopcheck was unavailable. |
</threat_model>
