<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| renderer -> preload -> IPC | User-entered adapter/path/source actions cross from untrusted renderer into main. |
| IPC -> source/scanner/cache services | Validated commands can change local app registry/cache state but must remain read-only toward harness/repo roots. |
| Electron main composition -> renderer bridge | Main-process dependencies must not leak broad filesystem, shell, or adapter-private authority. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-09-01 | Elevation of Privilege | `preload/index.ts` | mitigate | Expose only named source methods; no generic `ipcRenderer`, filesystem, shell, Electron, or invoke bridge. |
| T-03-09-02 | Tampering | `sources:*` IPC handlers | mitigate | Validate all requests with strict Zod schemas and restrict operations to local registry/cache state; no source-root mutation. |
| T-03-09-03 | Information Disclosure | IPC error responses | mitigate | Return sanitized failures only; no raw stack traces, raw filesystem dumps, raw events, or adapter-private object names. |
| T-03-09-04 | Repudiation | scan/validation IPC operations | mitigate | Keep validate and scan as distinct named methods whose responses persist validation, scan, cache, diagnostics, and timestamps. |
| T-03-09-SC | Tampering | npm/pip/cargo installs | mitigate | No package install is planned in this plan. If implementation attempts any install, add a blocking human verification checkpoint because slopcheck was unavailable. |
</threat_model>
