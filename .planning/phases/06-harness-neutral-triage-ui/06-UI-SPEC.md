# Phase 06 - UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Phase UI Scope

Phase 6 turns the current shell into a real read-only triage workspace. It must add harness-neutral `Overview`, `Projects`, `Sessions`, `Session Detail`, `Run Audit`, and `Diagnostics` surfaces that expose shared-core verification, audit, lifecycle, capability-gap, and diagnostic truth without introducing provider-specific behavior branches.

Primary visual anchors by surface:

- `Overview`: attention-first summary cards plus recent activity and recent sessions.
- `Projects`: selected project row and its truth panel, including explicit git/GitHub placeholders.
- `Sessions`: selected session row and a denser triage summary panel.
- `Session Detail`: sticky session summary rail plus chronological mixed-evidence timeline.
- `Run Audit`: sticky verdict rail plus grouped evidence sections.
- `Diagnostics`: grouped operator-style warning buckets with counts first and raw codes/messages second.

Keep V1 and Phase 7 boundaries intact. Do not add session launch/resume, approve/reject, terminal control, PR creation, branch/worktree cleanup, arbitrary shell execution, live watcher controls, repo mutation, real git/GitHub provider conclusions, provider-specific page branches, or dashboard math that converts missing evidence into `0`, `Passed`, or `Clean`.

Chronology and judgment stay separate. `Session Detail` owns the timeline. `Run Audit` owns grouped claim-vs-evidence review. `Sessions` remains the fast triage surface, not the full evidence browser.

## Spacing Scale

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge spacing, timeline marker gaps |
| sm | 8px | Filter-chip gaps, metadata clusters, compact control padding |
| md | 16px | Default element spacing, row padding, metric-card internals |
| lg | 24px | Route padding, panel padding, section spacing |
| xl | 32px | Split-pane gaps, page gutters, major chart/list separation |
| 2xl | 48px | Empty/error state offset, wide dashboard section breaks |

Exceptions: 44px minimum interactive hit area for nav items, filter chips, tabs, rows, and primary actions. The topbar remains 56px high. Sticky summary rails may use 72px heading bands when they include route title, status, and actions together.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F8FAFC` | App background, route canvas, dashboard breathing space |
| Secondary (30%) | `#FFFFFF` | Sidebar, topbar, list surfaces, detail panels, metric cards |
| Accent (10%) | `#2563EB` | Active nav item, selected row indicator, active filter chip, primary CTA, focus ring, explicit drill-in links |
| Destructive | `#DC2626` | Verification failed, load errors, blocking diagnostics, and future destructive actions only |

Accent reserved for: active route state, selected list row border/fill cue, `Reload Triage Data`, active harness/status filters, focused timeline anchors, and links that move the user from summary to detail. Do not use accent for generic counts, success states, placeholder git fields, passive metric cards, or decorative chart fills.

Supporting neutrals and semantic support:

| Token | Value | Usage |
|-------|-------|-------|
| Text strong | `#111827` | Primary readable text |
| Text muted | `#64748B` | Secondary labels, helper text, quiet metadata |
| Border | `#CBD5E1` | Panel boundaries, row separators, filter outlines, audit section dividers |
| Selected fill | `#EFF6FF` | Selected row background only |
| Neutral status fill | `#F1F5F9` | `Unknown`, `Unsupported`, `Not Run`, `Clean`, and placeholder git/GitHub states |
| Attention fill | `#FFF7ED` | `Needs Review`, `Cancelled`, `No Verification`, parser warnings, capability-gap summaries |
| Attention text | `#B45309` | Attention badges and grouped warning counts |

State color rule: never use green success styling in this phase. `Passed` and `Clean` remain explicit neutral badges so success does not visually outrank missing-evidence warnings. `Verification Failed` may use destructive red. `Needs Review`, `Cancelled`, `Not Run`, parser warnings, and capability gaps use attention styling or neutral badges, not accent blue.

## Layout Contract

| Area | Contract |
|------|----------|
| Window | Preserve the existing desktop shell baseline. Full triage density is optimized for 1100px by 720px and above, but the app must remain usable down to the current 800px by 680px minimum by stacking split panes and collapsing multi-column dashboards into single columns. |
| Shell chrome | Preserve the Phase 2/3 shell: 224px left sidebar, 56px topbar, secondary surfaces, one border divider, and no modal-first navigation. |
| Navigation | `Overview`, `Projects`, `Data Sources`, `Sessions`, and `Diagnostics` are real routes in Phase 6. `Overview` becomes the default landing route once triage routes exist. `Data Sources` remains available as the setup/config route, not the permanent home screen. |
| Routes | Recommended route map: `/overview`, `/projects`, `/data-sources`, `/sessions`, `/sessions/:sessionId`, `/sessions/:sessionId/audit`, and `/diagnostics`. `/` redirects to `/overview`. Unknown routes redirect to `/overview`. |
| Overview layout | Use three scan-first bands: top summary metrics, middle activity and attention split, bottom recent sessions/projects strip. Metric cards are compact bordered surfaces, not glossy hero tiles. Harness filters sit near the top and affect the whole page. |
| Projects layout | Use a split list/detail pattern matching the current shell language. Left list column is `minmax(360px, 42%)`; right truth panel is `minmax(0, 1fr)`. Project rows are dense and scannable, not card mosaics. |
| Sessions layout | Preserve the existing list/detail pattern but increase row density to support audit status, verification status, project, harness, counts, and warnings. Desktop uses `minmax(380px, 42%) minmax(0, 1fr)`. Below 960px, stack preview below the list. |
| Session Detail layout | Use two columns on desktop: sticky summary rail `minmax(280px, 320px)` and timeline content `minmax(0, 1fr)`. Below 1080px, stack the summary above the timeline. |
| Run Audit layout | Use a sticky verdict rail on the left or top plus grouped evidence sections on the right or below. Each evidence group is a bordered section with concise summary row, not a long undifferentiated feed. |
| Diagnostics layout | Use grouped sections by source area and severity. Lead with grouped counts and terse summaries, then show the underlying sanitized rows. Avoid raw JSON viewers and nested accordions deeper than one level. |
| Row treatment | Project rows and session rows remain bordered list rows with at least 88px height for projects and 96px height for sessions. Selected rows use a 4px accent border plus selected fill. |
| Metric cards | Metric cards use compact bordered surfaces with one display value, one label, and optional supporting note. Do not introduce oversized charts or glossy KPI dashboards. |
| Timeline entries | Timeline entries use a single vertical flow with event-type icon, timestamp, source label, and evidence summary. Messages, tool calls, shell commands, file mutations, output artifacts, lifecycle events, and unknown raw events share the same base skeleton. |
| Empty/error areas | Center within the route content area with `2xl` top offset. Use concise operator-style copy and one clear next step. No mascot illustrations, marketing prose, or ambient gradients. |

## State Vocabulary

Use these exact user-facing labels unless an upstream DTO supplies a more specific sanitized reason:

| Category | Labels |
|----------|--------|
| Run Audit | `Active`, `Cancelled`, `Verification Failed`, `Incomplete`, `Needs Review`, `Clean`, `Unknown` |
| Verification | `Passed`, `Failed`, `Not Run`, `Unsupported`, `Unknown` |
| Lifecycle | `Active`, `Completed`, `Cancelled`, `Unknown` |
| Capability truth | `Supported`, `Unsupported`, `Unknown` |
| Git/GitHub placeholders | `Unknown`, `Unsupported` |
| Diagnostics severity | `Info`, `Warning`, `Error` |

Truth rules:

- Numeric counts are shown only when the underlying evidence or capability is supported and present.
- When a metric depends on unsupported evidence, render the field as `Unsupported`, not `0`.
- When a metric depends on missing or indeterminate evidence, render the field as `Unknown`, not blank and not `0`.
- For project-level git and GitHub data in Phase 6, render explicit placeholders instead of hiding the fields. Use `Unknown` when the concept is expected later but not yet derived. Use `Unsupported` only when the current capability surface truly cannot represent the field.
- `No verification` is represented by `Not Run`, not `Passed`, `Clean`, or an empty cell.

## Interaction Contract

| Interaction | Contract |
|-------------|----------|
| Initial route | App lands on `Overview`. If no scanned session data exists yet, Overview shows a truthful setup empty state with a CTA to `Open Data Sources`. |
| Reload action | Use one route-level primary action label, `Reload Triage Data`, on Overview, Projects, Sessions, Session Detail, Run Audit, and Diagnostics. It re-requests sanitized view models only. It must not launch scans, mutate sources, or run shell commands. |
| Harness filters | Overview, Projects, Sessions, and Diagnostics expose harness filters driven by adapter metadata only. Filter chips preserve state while navigating among triage routes during the same app session. |
| Overview drill-in | Metric cards and recent-activity lists link outward to Projects, Sessions, Session Detail, or Run Audit. Do not embed full detail panels inside Overview. |
| Project selection | Clicking a project row updates the right-hand truth panel in place. The panel includes repo path, observed harnesses, latest activity, session count, latest verification/audit truth, and explicit git/GitHub placeholders. |
| Sessions selection | Clicking a session row updates a richer summary panel in place. Keyboard arrows move through rows and Enter selects the focused row, matching the existing Sessions route behavior. |
| Session drill-in | The selected session summary panel offers explicit actions `Open Session Detail` and `Open Run Audit`. Do not collapse these deeper views back into the list page. |
| Session Detail | Lead with harness badge, project, IDs, lifecycle, verification, run audit, and attention reasons. Timeline follows in chronological order with progressive disclosure for long evidence summaries and raw artifact references. |
| Run Audit grouping | Group evidence by `Claim vs Evidence`, `Verification`, `Files Changed`, `Commands`, `Cancellation / Incompletion`, `Git / GitHub`, `Capability Gaps`, and `Parser Diagnostics`. Empty groups show explicit no-evidence or unsupported states rather than disappearing silently. |
| Diagnostics | Diagnostics groups are filterable by severity and source area. Each row shows severity, code, concise message, adapter or source context, and related session or project label when available. Do not expose raw stacks, raw filesystem dumps, or unsanitized shell output. |
| Placeholder git fields | Project and session surfaces show branch, HEAD, dirty state, changed files, untracked files, and PR-related fields as explicit placeholders until Phase 7 providers exist. Never infer them from shell text or final-answer claims. |
| Capability warnings | Capability gaps appear in-page near the affected metric or section and also in a grouped warning area. Do not force the user to infer missing coverage from absent columns. |
| Timeline density | Timeline defaults to collapsed summaries for shell commands, tool calls, and file mutations, with inline expand/collapse behavior. Avoid modal raw-output viewers in Phase 6. |
| Empty states | Every route has a route-specific empty state with one next action. Empty routes do not redirect unexpectedly except the root redirect to Overview. |
| Read-only scope | No launch, stop, terminal input, approve, reject, PR, git mutation, source mutation, delete, reset, cleanup, or retry-with-command UI is allowed in Phase 6. |
